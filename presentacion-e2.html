Objetivo de la Entrega

El presente entregable constituye el Producto Mínimo Viable (MVP) operativo para la empresa TA-DIN. Su propósito es validar de extremo a extremo (End-to-End) la arquitectura técnica implementada, garantizando la correcta comunicación entre la interfaz de usuario, la lógica de negocio en el servidor y la persistencia de datos en el motor relacional, sentando las bases para la futura certificación de procesos bajo la norma ISO 9001.

Frontend Validado Backend Activo BD Conectada E2E Verificado ISO 9001
02

Requisitos por Capas — Página Web

A. Frontend — Interfaz y Navegación
  • Vistas Clave: Interfaz completamente codificada para Home y el Dashboard o catálogo principal, basadas en el mockup.
  • Navegación Fluida: Enrutamiento real y funcional entre las vistas sin recargas innecesarias de página.
B. Backend y Base de Datos
1. Estructura de Datos y Persistencia
  • Arquitectura estática: El proyecto web ha sido concebido bajo una arquitectura estática orientada a la presentación de información. No requiere motor de base de datos activa.
  • Ausencia de Tablas Relacionales: El objetivo exclusivo del sitio es la visualización de contenido y redirección mediante hipervínculos. Toda la información está estructurada directamente en HTML y CSS.
  • Despliegue e Infraestructura: Los archivos fuente fueron transferidos y desplegados en el hosting DonWeb mediante el cliente FTP FileZilla, operando bajo el dominio propio del proyecto.
2. Endpoints y Comunicación HTTP
  • Arquitectura Serverless/Static: El proyecto no expone una API REST ni posee endpoints operativos para lógica de negocio.
  • Resolución de Peticiones: El servidor en DonWeb solo despacha el documento estático frente a las peticiones HTTP. La navegación se resuelve a través del enrutamiento nativo del navegador hacia URLs externas o secciones internas de la página.
03

Requisitos por Capas — Reloj Biométrico

Interfaces del Dispositivo
1. Interfaz de Empleado (Marcación Rápida)
  • Pantalla de espera: Reloj digital en tiempo real y mensaje de instrucción para colocar la huella.
  • Confirmación: Al detectar la huella, muestra por 2 s la foto del empleado, nombre, hora e indicador visual (Verde = Éxito / Rojo = Error).
2. Panel de Administrador (Navegación Lateral)
  • Dashboard: Gráficos en tiempo real con el estado del personal (presentes, ausentes, llegadas tarde).
  • Gestión de Usuarios: Registro de empleados y enrolamiento de huellas.
  • Reportes: Filtros para consultar y exportar el historial de asistencia.
  • Configuración: Sincronización del dispositivo y horarios.
B. Backend y Base de Datos (ZKBioTime.Net)
Base de Datos Activa
  • Motor BD: SQLite por defecto · compatible con SQL Server, MySQL o PostgreSQL.
  • Tabla Usuarios: ID_Usuario, Nombre, Apellido, plantillas biométricas codificadas (huellas y rostros).
  • Tabla Registros: ID_Registro, ID_Usuario, FechaHora_Marcacion, Tipo_Movimiento (Entrada/Salida).
Endpoints y Protocolo de Comunicación
  • Protocolo estándar: Comunicación TCP/IP directa en la red local · Puerto 4370 (no REST/HTTP).
  • Transmisión: El software usa protocolo propietario ZKTeco para autenticar y descargar logs.
  • Alternativa técnica: SDK de ZKTeco en C# o Python para desarrollar API personalizada con endpoints HTTP propios.

Vista del Sistema

El software ZKBioTime.Net actúa como middleware oficial del dispositivo biométrico, gestionando de forma automática la estructura relacional de los datos de asistencia.

Se instala con base de datos local SQLite por defecto, con compatibilidad para motores más robustos como SQL Server, MySQL o PostgreSQL.

Panel de Administrador

El panel de administración de ZKBioTime.Net proporciona una vista centralizada del estado del personal en tiempo real: presentes, ausentes y llegadas tarde.

Desde aquí el administrador gestiona el registro de empleados, el enrolamiento de huellas y la exportación de reportes de asistencia.

06

Gestión de Usuarios y Enrolamiento

La gestión de usuarios permite registrar a cada empleado con sus datos personales (ID_Usuario, Nombre, Apellido) y vincularlos a sus plantillas biométricas codificadas.

El proceso de enrolamiento captura las huellas dactilares y datos de rostro, almacenándolos de forma segura en la base de datos del sistema.

07

Comunicación TCP/IP · Puerto 4370

La comunicación entre el reloj y el software se realiza por protocolo TCP/IP de forma directa en la red local, operando en el Puerto 4370, sin necesidad de endpoints HTTP convencionales.

El software envía peticiones empaquetadas usando el protocolo propietario de ZKTeco para autenticar al administrador y descargar el listado de logs de asistencia.
